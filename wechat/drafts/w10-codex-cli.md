一个多月前，我开始用一个叫 Codex CLI 的 AI 编程 Agent。

它不是一个代码补全工具，不是 GitHub Copilot。它是一个**能在终端里直接操作你的电脑**的 AI——创建文件、改代码、跑命令、发公众号、生成视频封面，全部自动完成。

这篇文章是一个月来的真实使用记录。没有 hype，只有实战。

---

## 一、我的使用场景：从算法动画到公众号全流程

我的工作流是这样的：

1. 从选题池挑一道 LeetCode 题
2. 写脚本（75-82 秒旁白）
3. 生成 TTS 语音（edge-tts）
4. 写 HTML 动画（GSAP + HyperFrames）
5. 渲染成 MP4 视频
6. 生成封面图
7. 发布到抖音和 B站
8. 同时生成公众号深度文章

过去这套流程我一个人做，一条视频至少要 4-5 小时。现在全部交给 Codex CLI——选题、脚本、动画代码、TTS、渲染、封面、公众号文章，一条命令触发，全自动跑。

生产效率提升了不是一个量级的问题，是工作流被彻底重构了。

---

## 二、Codex CLI 的三个核心能力

**1. 文件系统全权限操作**

它能创建目录、写文件、读文件、删文件、移动文件。这和 Copilot 的「在当前文件里补全」是两种完全不同的交互模型。Codex 是在操作你的整个项目，不是一个文件。

**2. Shell 命令执行**

它能执行终端命令并读取输出。`ffmpeg` 渲染视频、`bun` 跑 TypeScript 脚本、`python3` 生成封面图——所有这些它都能调用，而且会根据输出判断是否成功、出错了怎么修。

**3. 多步骤编排**

最关键的差异化能力。你可以说「写一篇公众号文章、生成封面、发到草稿箱」，它会自己拆成写稿→生图→调 API→验证→报告五个步骤，一步步执行。中间出错了会自动重试。

## 三、一个月踩过的五个坑

**坑 1：路径依赖幻觉**

AI 不知道你的项目结构。如果不显式告诉它文件在哪，它会猜测路径，然后报「file not found」。解决方法是给一个清晰的 AGENTS.md 文件，描述目录结构和命名规范。

**坑 2：一次做太多事会丢失上下文**

早期的用法是「帮我做完整个视频」。但上下文太长时，它在中间步骤就开始忘记前面的约束。后来学会了拆成小任务——先写脚本，确认无误；再生成动画，逐一检查。

**坑 3：生图 AI 不稳定**

让 Codex 调用豆包 Seedream 生成封面时，第一次往往不满意。最佳做法是先让它生成一版，你给反馈（「字体太小」「颜色太暗」），它再调整。两到三轮后效果稳定。

**坑 4：API 连接问题**

npm 源挂了、微信 API 限流了、豆包超时了——这些问题 Codex 会报错但不会主动换源或等待重试。需要在 AGENTS.md 里预设好备用方案。

**坑 5：代码质量需要审核**

对于算法动画的 HTML/JS 代码，AI 写的代码能跑但不够优化。比如 GSAP timeline 没用对、CSS 布局冗余。需要人工看一眼再执行。

## 四、四个最实用的技巧

**写一份详细的 AGENTS.md**

这是最重要的。里面放什么？目录结构、命名规范、已知高频模式、你的常用命令。AI 每次启动会读这个文件，相当于给了它一份「项目说明书」。

**一次只做一件事，分步确认**

不要说「做第 10 题」。说「写第 10 题的脚本」，确认脚本无误后「生成 TTS」，再确认后「写 HTML 动画」。每一步产出可验证的中间结果。

**预设模板和工具路径**

把常用工具（ffmpeg、bun、python3）的调用命令写进 AGENTS.md，让 AI 不用每次重新想参数。模板文件（视频 HTML 模板、公众号封面 prompt 模板）也放好路径。

**建立反馈循环**

AI 生成了内容后，告诉它哪里好哪里不好。下次它的输出会更接近你的审美。尤其生图和写文章这两类主观任务，反馈是关键。

---

## 五、AI Agent 的现在与未来

坦白说，现在的 AI Agent 还不能完全替代一个高级工程师。但在**已定义好工作流的重复性任务**上，它的效率碾压式领先。

关键不是 AI 有多聪明，而是你有没有把工作流梳理清楚。流程越清晰、约束越明确、反馈越及时，AI 就越像一个能独立工作的同事而不是一个需要手把手教的实习生。

我的 AGENTS.md 从最初的 20 行膨胀到了 200+ 行。投入越多的「教学成本」，产出越大。

**AI Agent 不是在取代程序员，是在解耦「想法」和「实现」。**
