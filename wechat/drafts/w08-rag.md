2024-2025 年，RAG（检索增强生成）成了 LLM 最主流的落地范式。逻辑很简单：大模型的知识有截止日期，还会幻觉。给它外挂一个知识库，需要时现查现用。

听起来靠谱。一万个团队都这么想。然后一落地，全卡在同一个问题上：**检索出来的文档不相关。** 不相关 → Prompt 错 → 回答错 → 用户骂 AI 傻。

不是模型的问题。是 RAG 的每一步都可能出错，而且错误会在流水线里放大。以下七条优化策略，覆盖从文档准备到生成回答的全链路。不需要全上，但需要知道每一条在解决什么问题。

---

## 一、文档切块：最重要也最容易被忽略

切成多大的块，是 RAG 系统最重要的超参数。没有之一。

块太大 → 噪声太多。用户问「今年 Q1 营收」，检索出一整篇 5000 字财务报告，LLM 自己都找不到关键数字。块太小 → 上下文断裂。「它」字被切到另一个块里去了，LLM 不知道「它」指代谁。

有四种主流策略：

- **固定大小**：最粗暴，每 500 字一刀切，块之间留 10% 重叠。简单但不够聪明
- **递归分块**：先按段落分割，太长按句子分，还长按标点分。LangChain 的默认方案，80% 的场景够用
- **从小到大**：同一段文档切成多种粒度（100 字 / 500 字 / 2000 字），全部存入向量库。检索时可以从小块开始匹配，再拉出大块给 LLM 看上下文。缺点是存储成本翻几倍
- **语义分块**：用模型判断「语义上应该切在这里」而不是机械分割。准确率最高，但需要额外的模型推理

**选择的原则**：一个理想的文本块，在脱离上下文后，对人类来说仍然应该有意义。如果连你都不知道这段在说什么，模型更不可能。

---

## 二、嵌入模型：检索的引擎不能省

块切好了，下一步是把文本转成向量。这个步骤的选择直接影响检索命中率。

OpenAI 的 `text-embedding-3-large`（3072 维）是目前通用场景的标杆。如果预算有限或想开源，BGE-M3（1024 维，多语言）是个不错的选择。

但有一个关键点：**嵌入模型和用户的问法要匹配。** 如果你的用户习惯用口语提问（比如「怎么退钱」），但知识库文档是正式表述（「售后政策」），检索就会漏。解决方法是选一个在口语-正式文的语义对齐上表现好的模型，或者用一批用户真实查询微调一下。

另一个容易忽略的：**多语言场景。** 用户用中文问，文档有英文的。BGE-M3 和 multilingual-e5 系列支持跨语言检索，不要拿单语言模型硬上。

---

## 三、查询重写：用户不会提问，你得帮他

现实中的用户提问是短句、口语化、缺少上下文。比如「上次那个 bug 怎么修了」，谁知道「上次」是哪个、「那个 bug」是哪个？

查询重写就是用 LLM 把用户粗糙的原始问题改写成适合检索的形式。三种常用手法：

- **子问题拆解**：「这个产品去年和今年对比」→ 拆成「2025 年销售额」「2024 年销售额」两个独立查询分别检索，结果再合并
- **假设文档（HyDE）**：让 LLM 先生成一段假设的答案文本，用这段假答案去检索。因为假答案和真实文档的语言风格更接近，匹配率比原始问题高
- **多轮补全**：对话场景下，把「它多少钱」→「iPhone 16 Pro Max 多少钱」，补全代词和省略

查询重写的 ROI 极高：多一次 LLM 调用，检索命中率可能提升 30-50%。

---

## 四、混合检索：语义不够，关键词来凑

向量检索（语义检索）擅长找「意思相近」的内容，但有两个致命盲区：

- **精确匹配**：用户搜「AK-47 参数」，向量可能返回「步枪发展史」而不是规格表
- **专有名词**：产品型号、错误代码、人名，向量模型多半没见过

混合检索（Hybrid Search）同时跑向量检索和 BM25 关键词检索，两边的结果用 RRF（倒数排名融合）合并去重。精确匹配交给关键词，语义理解交给向量。生产环境的 RAG 系统几乎都这么做。

---

## 五、重排序：最相似的 ≠ 最相关的

向量检索返回 Top-20，语义都很相似。但「语义上相似」不等于「能回答用户问题」。

用户问「最新上映的科幻电影」，检索可能返回「科幻电影历史回顾」——语义高度相关，但时间错位，毫无帮助。

重排序（Re-ranking）用更精细的 cross-encoder 模型对初选结果做二次打分。不仅看语义，还看查询意图、上下文、时效性。目前最常用的是 Cohere Rerank 和 BGE-Reranker，都能把 Top-20 里真正有用的文档推到前 5。

这一招往往是把 RAG 从 80 分拉到 90 分的关键。

---

## 六、Self-RAG：让 AI 自己检查自己

传统 RAG 是单向流水线：检索 → 生成 → 输出。错了就错了。

Self-RAG 加了一个「反思」环节。生成答案后，LLM 自己评估检索到的文档是不是真正相关、答案是否有依据。如果评估不通过，重新检索或调整生成策略。

Scoring tokens 是实现方式之一：模型在生成答案的同时，插入特殊的反思标记（<relevant> / <irrelevant> / <fully supported> / <partially supported>），然后根据这些标记决定是输出还是重试。

这多跑几轮调用，但对于对准确率有要求的场景（医疗、法律、金融），多花的几秒完全值得。

---

## 七、提示词：最后一厘米决定了全部

RAG 的所有优化最终都汇聚在一条 Prompt 上。这里写砸了，前面全白费。

RAG Prompt 有一条黄金法则：**明确告诉 LLM，只根据检索到的文档回答，不要用你自己的知识。**

```
你是一个客服助手。请只根据以下上下文回答。
如果上下文中没有相关信息，直接说「我目前没有相关信息」，
不要自己编造。

上下文：
{检索结果}

用户问题：
{用户输入}
```

还有一个细节：**标明来源。** 在 Prompt 里要求 LLM 在回答中标注引用了哪段文档。这不仅让用户能核实信息，也倒逼 LLM 更谨慎地使用检索结果。

这两句约束——「不知道就说不知道」「标明来源」——能砍掉 80% 的幻觉，比任何复杂优化都立竿见影。

---

没有银弹。七条策略各解决一个环节的问题。但如果你只做一件事，从第一条开始：把文档切对了。块切不好，后面的一切都是浪费。
